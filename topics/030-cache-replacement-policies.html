Cache Replacement Policies

⏱ 5 min read 🟡 Intermediate Caching Algorithms

Core Concept

When implementing caching, it’s important to have a cache replacement policy to determine which items in the cache should be removed when the cache becomes full. Here are some of the most common cache replacement policies:

1. Least Recently Used (LRU)

LRU is a cache replacement policy that removes the least recently used item from the cache when it becomes full. This policy assumes that items that have been accessed more recently are more likely to be accessed again in the future.

2. Least Frequently Used (LFU)

LFU is a cache replacement policy that removes the least frequently used item from the cache when it becomes full. This policy assumes that items that have been accessed more frequently are more likely to be accessed again in the future.

3. First In, First Out (FIFO)

FIFO is a cache replacement policy that removes the oldest item from the cache when it becomes full. This policy assumes that the oldest items in the cache are the least likely to be accessed again in the future.

4. Random Replacement

Random replacement is a cache replacement policy that removes a random item from the cache when it becomes full. This policy doesn’t make any assumptions about the likelihood of future access and can be useful when the access pattern is unpredictable.

Comparison of different replacement policies

Each cache replacement policy has its advantages and disadvantages, and the best policy to use depends on the specific use case. LRU and LFU are generally more effective than FIFO and random replacement since they take into account the access pattern of the cache. However, LRU and LFU can be more expensive to implement since they require maintaining additional data structures to track access patterns. FIFO and random replacement are simpler to implement but may not be as effective in optimizing cache performance. Overall, the cache replacement policy used should be chosen carefully to balance the trade-off between performance and complexity.

Beginner Analogy

💡
Cache replacement policies are like deciding which books to keep on a small shelf when it gets full. LRU (Least Recently Used) removes the book you haven't opened in the longest time. LFU (Least Frequently Used) removes the book you've opened the fewest times overall. FIFO (First In, First Out) removes whichever book has been on the shelf the longest, regardless of use. Each strategy makes a different bet about which content will be needed next.

Real-World Examples

🔴 Redis
Redis supports multiple eviction policies including allkeys-lru, volatile-lru, allkeys-lfu, and noeviction. Production deployments choose based on access patterns: LRU works well for time-sensitive data like sessions, while LFU is better for stable popular data like product catalogs. Redis tracks access frequency using a probabilistic counter to keep LFU efficient.
🔍 Google
Google's web search index uses sophisticated variants of LFU for caching query results, favoring results for high-traffic search terms. An obscure long-tail query is evicted before a common popular query even if the common query was cached more recently. This maximizes the cache hit rate for the majority of user traffic.
🏠 Airbnb
Airbnb uses LRU caching for property listing data, reasoning that recently viewed listings are more likely to be viewed again during an active search session. TTL-based expiration ensures that even LRU-retained items don't become stale. Combining LRU with TTL gives both recency-based eviction and freshness guarantees.

Interview Tips

  • 💡 Know the three main policies cold: LRU evicts the least recently accessed item, LFU evicts the least frequently accessed item, and FIFO evicts the oldest-inserted item.
  • 💡 Discuss the LRU implementation: it requires a doubly linked list plus a hash map to achieve O(1) get and put — this is a classic interview question in itself.
  • 💡 Explain when LFU outperforms LRU: when some items are accessed very frequently over a long period and LRU would evict them during a burst of one-time requests.
  • 💡 Mention that many production caches (like Redis) combine policies — for example, LRU with TTL ensures both recency-based eviction and time-based expiration.

Key Takeaways

✅ LRU evicts the item that was least recently accessed and is the most widely used replacement policy.
✅ LFU evicts the item that has been accessed least often overall — better when popularity is stable.
✅ FIFO is simple but often suboptimal because it ignores access patterns.
✅ TTL (Time-To-Live) is a complementary mechanism that expires items after a set time regardless of access.
✅ LRU can be implemented in O(1) time using a hash map paired with a doubly linked list.
