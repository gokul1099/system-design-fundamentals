What is Heartbeat

вҸұ 4 min read рҹҹў Beginner Heartbeat Failure Detection Distributed Systems

Core Concept

Background

In a distributed environment, work/data is distributed among servers. To efficiently route requests in such a setup, servers need to know what other servers are part of the system. Furthermore, servers should know if other servers are alive and working. In a decentralized system, whenever a request arrives at a server, the server should have enough information to decide which server is responsible for entertaining that request. This makes the timely detection of server failure an important task, which also enables the system to take corrective actions and move the data/work to another healthy server and stop the environment from further deterioration.

Solution

Each server periodically sends a heartbeat message to a central monitoring server or other servers in the system to show that it is still alive and functioning.

Heartbeating is one of the mechanisms for detecting failures in a distributed system. If there is a central server, all servers periodically send a heartbeat message to it. If there is no central server, all servers randomly choose a set of servers and send them a heartbeat message every few seconds. This way, if no heartbeat message is received from a server for a while, the system can suspect that the server might have crashed. If there is no heartbeat within a configured timeout period, the system can conclude that the server is not alive anymore and stop sending requests to it and start working on its replacement.

Think of a heartbeat like a friend who texts you 'I'm alive' every 10 minutes. If you stop getting texts, you assume something went wrong. Distributed systems work the same way вҖ” nodes send tiny signals to prove they're still running, and the cluster acts if signals stop arriving.

Real-World Examples

рҹҗҳ Apache ZooKeeper
ZooKeeper uses heartbeat sessions. If a client doesn't renew its session within the timeout, ZooKeeper treats it as failed and releases its ephemeral nodes, triggering leader re-election.
вҳёпёҸ Kubernetes
The kubelet on each node sends heartbeats to the control plane every few seconds. If the control plane misses heartbeats for node-monitor-grace-period, it marks the node NotReady and reschedules pods.
вҳҒпёҸ AWS
AWS Elastic Load Balancer performs health checks (a form of heartbeat) against backend instances. If an instance fails a configurable number of consecutive checks, ELB stops routing traffic to it.

Interview Tips

  • рҹ’Ў Mention the heartbeat interval vs timeout trade-off: short intervals detect failures faster but add network overhead.
  • рҹ’Ў Discuss false positives вҖ” a slow network can look like a dead node; mention grace periods to reduce false alarms.
  • рҹ’Ў Heartbeats are the basis of leader election: if the leader stops heartbeating, followers start an election.

Key Takeaways

вң… Heartbeats are periodic signals that prove a node is alive.
вң… If heartbeats stop arriving, the system assumes failure and triggers recovery.
вң… The timeout value is a trade-off between fast failure detection and false positives.
вң… Used in ZooKeeper, Kubernetes, Consul, and virtually every distributed system.
