NoSQL Databases

⏱ 4 min read 🟡 Intermediate NoSQL Databases

Core Concept

NoSQL databases, also known as "Not Only SQL" databases, are a diverse group of non-relational databases designed to address the limitations of traditional SQL databases, particularly in terms of scalability, flexibility, and performance under specific workloads. NoSQL databases do not adhere to the relational model and typically do not use SQL as their primary query language. Instead, they employ various data models and query languages, depending on the specific type of NoSQL database being used.

The key characteristics of NoSQL databases include their schema-less design, which allows for greater flexibility in handling data; horizontal scalability, which makes it easier to distribute data across multiple servers; and their ability to perform well under specific workloads, such as high write loads or large-scale data storage and retrieval.

Types of NoSQL Databases

NoSQL databases can be broadly categorized into the following six types, each with its unique data model and use cases:

1. Key-value databases

Key-value databases store data as key-value pairs, where the key is a unique identifier and the value is the associated data. These databases excel in scenarios requiring high write and read performance for simple data models, such as session management and real-time analytics.

Use cases: Session management, user preferences, and product recommendations.

Examples: Amazon DynamoDB (in the cloud), Azure Cosmos DB, Redis.

2. In-memory key-value databases

The data is primarily stored in memory, unlike disk-based databases. By eliminating disk access, these databases enable minimal response times. Because all data is stored in main memory, in-memory databases risk losing data upon a process or server failure. In-memory databases can persist data on disks by storing each operation in a log or by taking snapshots.

Examples: Redis, Memcached, Amazon Elasticache.

Beginner Analogy

💡
NoSQL databases are like different kinds of filing systems for different needs. A document store is like a filing cabinet of folders — each folder can contain anything and its own unique set of documents. A key-value store is like a coat check — you hand in an item, get a ticket number, and retrieve exactly your item by that number. A column-family store is like a spreadsheet where most cells are intentionally left empty. A graph database is like a social network map where the connections between people matter as much as the people themselves.

Real-World Examples

📘 Facebook/Meta
Facebook uses multiple NoSQL systems: Cassandra (column-family) for inbox search and messaging, TAO (a graph-like store) for the social graph, and Memcached (key-value) for caching. Each NoSQL type was chosen because its data model perfectly matched the access pattern of that feature.
💼 LinkedIn
LinkedIn uses a graph database concept for its connection recommendations — knowing that Alice is connected to Bob who is connected to Carol is a graph problem, and graph databases make traversing these relationships orders of magnitude faster than SQL JOINs across millions of users.
🚗 Uber
Uber uses a document store (MongoDB) for certain microservices where schema flexibility matters — trip data can have varying attributes depending on product type (UberX, UberPool, Eats), and a document model accommodates these varying shapes without schema migrations.

Interview Tips

  • 💡 Know all four NoSQL types cold: document (MongoDB), key-value (Redis/DynamoDB), column-family (Cassandra/HBase), and graph (Neo4j) — interviewers test breadth here.
  • 💡 For each type, immediately associate a canonical use case: key-value for caching, document for user profiles, column-family for time-series data, graph for recommendations.
  • 💡 Point out that Redis serves double duty as both a key-value store and an in-memory cache — it's one of the most versatile tools in any system design interview.
  • 💡 When choosing a NoSQL database, the deciding factor is always the access pattern: how will you query this data? Key-based? Range queries? Traversals? The answer determines the type.

Key Takeaways

✅ NoSQL encompasses four main types: document, key-value, column-family, and graph databases, each with distinct data models.
✅ Document stores like MongoDB allow flexible, nested JSON-like records with no fixed schema.
✅ Key-value stores like Redis provide O(1) lookup by key, making them ideal for caching and session storage.
✅ Column-family stores like Cassandra excel at write-heavy time-series workloads with sparse, wide rows.
✅ Graph databases like Neo4j are optimized for relationship traversal queries that would require expensive JOINs in SQL.
