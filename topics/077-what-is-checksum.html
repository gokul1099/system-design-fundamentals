What is Checksum

⏱ 4 min read 🟢 Beginner Checksum Data Integrity Reliability

Core Concept

Background

In a distributed system, while moving data between components, it is possible that the data fetched from a node may arrive corrupted. This corruption can occur because of faults in a storage device, network, software, etc. How can a distributed system ensure data integrity, so that the client receives an error instead of corrupt data?

Solution

Calculate a checksum and store it with data.

To calculate a checksum, a cryptographic hash function like MD5, SHA-1, SHA-256, or SHA-512 is used. The hash function takes the input data and produces a string (containing letters and numbers) of fixed length; this string is called the checksum.

When a system is storing some data, it computes a checksum of the data and stores the checksum with the data. When a client retrieves data, it verifies that the data it received from the server matches the checksum stored. If not, then the client can opt to retrieve that data from another replica.

Beginner Analogy

💡
A checksum is like a receipt total at the bottom of a grocery bill. After scanning all items, the register adds them up and prints the total. When you review the receipt, you re-add the items yourself. If your total matches, nothing was missed. If it doesn't, something went wrong.

Real-World Examples

🐙 Git
Every Git commit, tree, and blob is identified by a SHA-1 (now SHA-256) checksum of its contents. If a single byte changes, the hash changes entirely — making corruption or tampering immediately detectable.
☁️ Amazon S3
S3 computes an MD5 checksum of every object. When you upload a file with the Content-MD5 header, S3 verifies the upload is uncorrupted before accepting it.
🌐 TCP/IP
Every TCP segment contains a 16-bit checksum computed over the header and data. Receiving hosts verify it; corrupted packets are discarded and retransmission is requested.

Interview Tips

  • 💡 Explain that checksums detect corruption but don't fix it — they trigger retransmission or rejection.
  • 💡 Distinguish checksum (fast, simple, not cryptographic) from hash (e.g. SHA-256, collision-resistant) from HMAC (tamper-proof with a key).
  • 💡 Mention where checksums appear: TCP, disk I/O, file transfer, database pages.

Key Takeaways

✅ A checksum is a small value computed from data used to detect accidental corruption.
✅ If the recomputed checksum doesn't match, the data was modified in transit or storage.
✅ Checksums are fast but not cryptographically secure — use SHA-256 or HMAC for security.
✅ Used in TCP/IP, Git, S3, and virtually every storage and transport system.
