What are Indexes

⏱ 4 min read 🟢 Beginner Indexes Databases Performance

Core Concept

Indexes are well known when it comes to databases. Sooner or later there comes a time when database performance is no longer satisfactory. One of the very first things you should turn to when that happens is database indexing.

The goal of creating an index on a particular table in a database is to make it faster to search through the table and find the row or rows that we want. Indexes can be created using one or more columns of a database table, providing the basis for both rapid random lookups and efficient access of ordered records.

Example: A library catalog

A library catalog is a register that contains the list of books found in a library. The catalog is organized like a database table generally with four columns: book title, writer, subject, and date of publication. There are usually two such catalogs: one sorted by the book title and one sorted by the writer name. That way, you can either think of a writer you want to read and then look through their books or look up a specific book title you know you want to read in case you don’t know the writer’s name. These catalogs are like indexes for the database of books. They provide a sorted list of data that is easily searchable by relevant information.

Simply saying, an index is a data structure that can be perceived as a table of contents that points us to the location where actual data lives. So when we create an index on a column of a table, we store that column and a pointer to the whole row in the index. Beginner Analogy

💡
A database index is like the index at the back of a textbook. Without it, finding all mentions of 'binary search' requires reading every page from cover to cover. With the index, you look up 'binary search,' find 'pages 47, 112, 203,' and jump directly to those pages. The index takes up extra space and must be updated every time the book is revised, but the speed benefit for lookup-heavy use is enormous.

Real-World Examples

🔍 Google
Google's search infrastructure is built on inverted indexes — for every word on the web, an index maps that word to all pages containing it. Without this index, searching the web would require scanning every page for each query, which is computationally impossible at scale. The entire premise of web search depends on indexing.
🏠 Airbnb
Airbnb indexes its listings on location, price, and availability. When a user searches for 'Paris listings under $100 available next weekend,' the database uses these indexes to quickly filter millions of listings down to hundreds of matches, rather than scanning every row.
💼 LinkedIn
LinkedIn indexes its member database on skills, location, and industry to power the 'People You May Know' and search features. Without indexes on these columns, a query like 'software engineers in San Francisco' would require a full table scan across 900 million member profiles.

Interview Tips

  • 💡 Always mention the tradeoff: indexes speed up reads but slow down writes (every insert/update must also update the index) and consume additional storage.
  • 💡 Explain that a clustered index physically reorders table rows by index key — there can only be one per table (usually the primary key), and it determines the row storage order.
  • 💡 Non-clustered indexes store a separate structure pointing back to row locations — you can have many per table, but each adds write overhead.
  • 💡 Mention index selectivity: an index on a boolean column (true/false) has low selectivity and is largely useless; an index on email addresses has high selectivity and dramatically reduces scan size.

Key Takeaways

✅ An index is a separate data structure that enables fast lookup of rows without scanning the entire table.
✅ Indexes trade write performance and storage space for dramatically faster read queries.
✅ The primary key is automatically indexed in most databases, providing O(log N) lookup by ID.
✅ Index selectivity measures how well an index narrows the search — high-cardinality columns (emails, IDs) benefit most from indexing.
✅ Clustered indexes determine physical row storage order; non-clustered indexes are separate lookup structures pointing to row locations.
