Types of Indexes

⏱ 4 min read 🟡 Intermediate Indexes Databases Performance

Core Concept

Database indexes are designed to improve the speed and efficiency of data retrieval operations. They function by maintaining a separate structure that points to the rows in a table, allowing the database to look up data more quickly without scanning the entire table.

There are various types of database indexes, each with its unique characteristics and use cases. Understanding these different index types is crucial for optimizing the performance of database systems and ensuring efficient data retrieval.

In this section, we will explore several common types of database indexes.

Beginner Analogy

💡
Different index types are like different kinds of reference tools. A B-tree index is like a sorted card catalog — you can find exact entries or browse ranges alphabetically. A hash index is like a numbered locker system — you instantly know locker 2847 from a combination, but you can't browse 'all lockers in the 2000s.' A full-text index is like a book's word index — it helps you find documents containing specific words but doesn't help with numeric ranges. A composite index is like a catalog sorted first by author, then by title — incredibly fast for 'author + title' searches, but the author part must come first.

Real-World Examples

🌳 PostgreSQL/Airbnb
Airbnb uses B-tree indexes on listing price and date fields, which support range queries like 'price BETWEEN 50 AND 150' or 'available_date > tomorrow.' B-tree indexes maintain sorted order, making them the best choice when queries involve comparison operators and range scans.
⚡ Redis
Redis's hash data structure underpins its key-value lookup — the hash function maps any key string to an O(1) bucket lookup. This is pure hash indexing: exact-match lookups are instant, but you cannot ask Redis for 'all keys starting with user:' efficiently without a separate index structure.
🔍 Google
Google Search uses inverted full-text indexes — for the query 'distributed systems,' the full-text index finds all documents containing both words instantly. Elasticsearch, widely used at LinkedIn, Twitter, and Cloudflare, is built entirely around full-text inverted indexes for search functionality.

Interview Tips

  • 💡 Know that B-tree is the default index in PostgreSQL and MySQL — it supports equality, range, and sort operations efficiently and is the right choice for most situations.
  • 💡 Explain the composite index prefix rule: an index on (last_name, first_name) accelerates queries filtering by last_name or last_name + first_name, but NOT queries filtering only by first_name.
  • 💡 Hash indexes are O(1) for equality lookups but cannot support range queries or sorting — mention this constraint when discussing hash index use cases.
  • 💡 Full-text indexes use inverted index structures to map words to documents — they're the foundation of any search feature and require specialized query syntax (MATCH/AGAINST or similar).

Key Takeaways

✅ B-tree indexes are the universal default, supporting range queries, equality, and sorting in O(log N) time.
✅ Hash indexes provide O(1) exact-match lookups but cannot support range queries, making them suitable only for equality comparisons.
✅ Full-text indexes use inverted structures to enable efficient text search across document content.
✅ Composite indexes cover multiple columns, but query filters must use the leftmost prefix of the indexed columns to benefit from the index.
✅ Choosing the right index type requires understanding the query patterns — range queries need B-trees, text search needs full-text, and pure key lookups can use hash indexes.
