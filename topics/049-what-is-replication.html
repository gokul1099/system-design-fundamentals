What is Replication

⏱ 4 min read 🟢 Beginner Replication Databases Distributed Systems

Core Concept

Database replication is the process of copying and synchronizing data from one database to one or more additional databases. This is commonly used in distributed systems where multiple copies of the same data are required to ensure data availability, fault tolerance, and scalability.

Beginner Analogy

💡
Replication is like making photocopies of an important document and storing them in different filing cabinets across the office. If one cabinet gets damaged, you still have the document in the others. But unlike physical copies that are made once and forgotten, database replication keeps all copies continuously synchronized as the original document changes. The more copies you have and the more spread out they are, the safer and more accessible your data becomes.

Real-World Examples

🐙 GitHub
GitHub replicates Git repositories across multiple geographic regions to ensure that if one data center fails, repositories remain accessible from replicas elsewhere. Repository data is written to primary storage and asynchronously replicated to secondary locations. This means millions of developers can continue working even during a regional AWS outage.
🎬 Netflix
Netflix replicates its Cassandra database across three AWS availability zones for every region. A write to Netflix's systems is only acknowledged after it has been written to two of the three replicas (quorum write). This provides durability — even if one AZ experiences a complete failure, no data is lost and reads continue from the remaining replicas.

Interview Tips

  • 💡 Define replication: maintaining copies of the same data on multiple nodes to provide fault tolerance, read scalability, and geographic distribution.
  • 💡 Distinguish replication from backup: replicas are live, continuously synchronized copies for operational use; backups are point-in-time snapshots for disaster recovery.
  • 💡 Explain the read-scaling benefit: by directing read traffic to replicas, the primary database handles only writes, dramatically increasing total read throughput.
  • 💡 Discuss the consistency challenge: asynchronous replication means replicas may lag behind the primary, causing stale reads — synchronous replication eliminates this but adds write latency.

Key Takeaways

✅ Replication maintains multiple synchronized copies of data across different nodes or locations.
✅ Replicas provide fault tolerance — if one node fails, others continue serving data.
✅ Read replicas offload read traffic from the primary, scaling total read throughput horizontally.
✅ Synchronous replication ensures no data loss but adds write latency; asynchronous replication is faster but risks losing recent writes on failure.
✅ Geographic replication reduces read latency for global users and provides disaster recovery across regions.
