Introduction to Caching

⏱ 4 min read 🟢 Beginner Caching Performance

Core Concept

What is Caching?

The cache is a high-speed storage layer that sits between the application and the original source of the data, such as a database, a file system, or a remote web service. When data is requested by the application, it is first checked in the cache. If the data is found in the cache, it is returned to the application. If the data is not found in the cache, it is retrieved from its original source, stored in the cache for future use, and returned to the application.

Caching can be used for various types of data, such as web pages, database queries, API responses, images, and videos. The goal of caching is to reduce the number of times data needs to be fetched from its original source, which can result in faster processing and reduced latency.

Caching can be implemented in various ways, including in-memory caching, disk caching, database caching, and CDN caching. In-memory caching stores data in the main memory of the computer, which is faster to access than disk storage. Disk caching stores data on the hard disk, which is slower than main memory but faster than retrieving data from a remote source. Database caching stores frequently accessed data in the database itself, reducing the need to access external storage. CDN caching stores data on a distributed network of servers, reducing the latency of accessing data from remote locations.

Key terminology and concepts

1. Cache: A temporary storage location for data or computation results, typically designed for fast access and retrieval.

2. Cache hit: When a requested data item or computation result is found in the cache.

3. Cache miss: When a requested data item or computation result is not found in the cache and needs to be fetched from the original data source or recalculated.

4. Cache eviction: The process of removing data from the cache, typically to make room for new data or based on a predefined cache eviction policy.

5. Cache staleness: When the data in the cache is outdated compared to the original data source.

Beginner Analogy

💡
Caching is like keeping your most-used tools on your workbench instead of walking to the storage room every time. The storage room has everything, but it takes time to get there. By keeping frequently needed tools within arm's reach, you finish jobs much faster. The trade-off is that your workbench has limited space, so you must choose what to keep handy.

Real-World Examples

👍 Facebook/Meta
Facebook uses Memcached at a massive scale to cache social graph data, news feed items, and user profile information. Without caching, every page load would require thousands of database queries. Their Memcached deployment handles billions of requests per second across thousands of servers.
🐦 Twitter/X
Twitter caches timelines in Redis to avoid recomputing the tweet feed for every request. When a user logs in, their pre-computed timeline is served from cache in milliseconds. This is critical given the read-heavy nature of timeline consumption.

Interview Tips

  • 💡 Always start by defining what caching is and why it matters — storing copies of expensive-to-compute or frequently-accessed data closer to the consumer.
  • 💡 Discuss the cache hit ratio: a high hit ratio means most requests are served from cache, avoiding expensive backend calls.
  • 💡 Mention different caching layers: CPU caches, browser caches, CDN caches, application-level caches, and database query caches.
  • 💡 Be ready to discuss when NOT to cache — data that changes frequently or must always be consistent across all users.

Key Takeaways

✅ Caching stores frequently accessed data in a fast, temporary storage layer to reduce latency and backend load.
✅ A cache hit returns data without touching the source; a cache miss requires fetching from the origin and storing in cache.
✅ Caching is most effective for read-heavy workloads with data that doesn't change too frequently.
✅ Every caching strategy involves a trade-off between freshness (consistency) and performance.
✅ Redis and Memcached are the two most widely used distributed caching systems in production.
