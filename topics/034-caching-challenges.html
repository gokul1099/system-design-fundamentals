Caching Challenges

⏱ 4 min read 🟡 Intermediate Caching Problems

Core Concept

Cache-related problems are a set of challenges that arise when implementing and managing caching systems in software applications. Here are the top cache-related problems and their possible workarounds:

1. Thundering Herd

The thundering herd problem occurs when a popular piece of data expires from the cache, leading to a sudden surge in requests to the origin server to fetch the missing data. This can cause excessive load on the origin server and degrade performance. Solutions to the thundering herd problem include using staggered expiration times, implementing a cache lock, or using background updates to refresh the cache before the data expires.

2. Cache Penetration

Cache penetration refers to the situation where requests for data bypass the cache and directly access the origin server, reducing the benefits of caching. This typically happens when requests are made for non-existent or rarely accessed data. To mitigate cache penetration, negative caching (caching negative responses) or using a bloom filter to check for the existence of data before querying the cache can be employed.

3. Big Key

A big key is a large piece of data that consumes a significant portion of the cache's capacity. Storing big keys can lead to cache evictions, reducing the overall effectiveness of the caching system. Solutions for handling big keys include compressing the data before caching, breaking the data into smaller chunks, or using a separate caching strategy specifically designed for large objects.

4. Hot Key

A hot key refers to a piece of data that is frequently accessed, causing contention and performance issues in the caching system. Hot keys can lead to cache thrashing and an unbalanced distribution of load. Solutions for dealing with hot keys include using consistent hashing to distribute the load more evenly, replicating the hot key across multiple cache nodes, or implementing a load balancing strategy to distribute requests across multiple instances of the hot key.

5. Cache Stampede (or Dogpile)

Cache stampede occurs when multiple requests for the same data are made simultaneously, causing excessive load on the cache and the origin server. Cache stampede can be addressed using techniques such as request coalescing (combining multiple requests for the same data into a single request) or implementing a read-through cache, where the cache itself fetches the missing data from the origin server.

Beginner Analogy

💡
Cache stampede is like a popular bakery that sells out of bread every day at noon. Everyone in the office knows the bread will be gone by 12:05, so at exactly noon, hundreds of people all rush to the bakery at once. The stampede overwhelms the counter staff just as they're trying to restock. Thundering herd is the same problem — when one popular resource expires, all waiting requestors charge the backend simultaneously.

Real-World Examples

🤖 Reddit
Reddit experienced cache stampedes when a highly-trafficked post's cache entry expired — thousands of concurrent requests all found a cache miss and simultaneously hit the database. The resulting database overload caused cascading failures. They mitigated this using mutex locking to allow only one request to repopulate the cache.
☁️ Cloudflare
Cloudflare handles thundering herd at the CDN level using 'request coalescing' — when multiple requests for the same uncached resource arrive simultaneously, only one request is forwarded to the origin while the others wait. The single origin response is then distributed to all waiting clients, preventing stampede on origin servers.
👍 Facebook/Meta
Facebook prevents cache poisoning by validating all data before writing to Memcached. If a malicious or malformed response were cached, it would be served to millions of users. Their cache write path includes integrity checks to ensure only valid data enters the cache.

Interview Tips

  • 💡 Explain cache stampede clearly: it occurs when a popular cached item expires and all concurrent requestors simultaneously hit the backend, causing a traffic spike.
  • 💡 Describe the mutex lock (or 'dog-pile lock') solution: only one thread/process is allowed to repopulate the cache; others serve stale data or wait.
  • 💡 Mention probabilistic early expiration: cache items are randomly expired slightly before their actual TTL, spreading the repopulation across time rather than at a single instant.
  • 💡 Discuss cache poisoning separately: it is a security concern where malicious data is inserted into the cache and then served to legitimate users — validation on write is the primary defense.

Key Takeaways

✅ Cache stampede (thundering herd) occurs when many requests simultaneously find a cache miss and overwhelm the backend.
✅ Mutex locking ensures only one process repopulates a cache entry while others wait or use stale data.
✅ Request coalescing at the proxy layer forwards only one backend request for simultaneous cache misses.
✅ Cache poisoning occurs when malicious or corrupted data is stored in the cache and served to users.
✅ Probabilistic early expiration spreads cache refresh load over time instead of creating a sudden spike.
