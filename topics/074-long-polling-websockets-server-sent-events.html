Long-Polling, WebSockets, and Server-Sent Events

⏱ 4 min read 🟡 Intermediate WebSockets SSE Long Polling Real-time

Core Concept

Long-Polling, WebSockets, and Server-Sent Events are popular communication protocols between a client like a web browser and a web server. First, let’s start with understanding what a standard HTTP web request looks like. Following are a sequence of events for regular HTTP request:

  1. The client opens a connection and requests data from the server.
  2. The server calculates the response.
  3. The server sends the response back to the client on the opened request.

Beginner Analogy

💡
Long-polling is like calling a restaurant to ask if your table is ready — you call, they say 'not yet, stay on hold,' and they tell you when it's ready, then you hang up and call again for next time. WebSockets is like being given a walkie-talkie — you and the restaurant can talk any time, in either direction, without the overhead of placing a new call. Server-Sent Events is like subscribing to a radio station — the broadcast flows to you continuously, but you can't broadcast back on that same channel.

Real-World Examples

💬 Discord
Discord uses WebSockets for real-time chat messaging — messages must flow bidirectionally (users send and receive) with sub-second latency. WebSockets maintain a persistent TCP connection that allows either party to send data at any time without the overhead of HTTP request-response cycles.
🐦 Twitter/X
Twitter's public streaming API uses Server-Sent Events for the 'new tweet' notification feed — tweets flow from server to client in real-time, but the client doesn't need to send messages back on the same connection. SSE's unidirectional nature is perfectly matched to this one-way push pattern.
💳 Stripe
Stripe's webhook dashboard uses long-polling for customers checking payment event status in cases where webhooks have not been set up. The client polls Stripe's server every 30 seconds, and the server holds the connection open until an event occurs or a timeout is reached — simpler than WebSockets for infrequent updates.

Interview Tips

  • 💡 Default to WebSockets for any bidirectional real-time communication requirement (chat, collaborative editing, multiplayer games) — it's the most efficient persistent connection protocol.
  • 💡 Recommend SSE for server-to-client push scenarios (notifications, live feeds, dashboards) where the client doesn't need to send data back — SSE is simpler and has built-in reconnection logic.
  • 💡 Use long-polling only as a fallback for environments where WebSockets are blocked by corporate firewalls or when updates are infrequent — it's the least efficient of the three.
  • 💡 Mention that WebSockets require a connection upgrade handshake starting with HTTP, and that connection state must be managed server-side, which complicates load balancing (sticky sessions or shared state like Redis Pub/Sub are needed).

Key Takeaways

✅ Long-polling simulates push by having the server hold an HTTP request open until data is available or a timeout occurs, then the client immediately re-polls.
✅ WebSockets establish a persistent, full-duplex TCP connection enabling low-latency bidirectional communication without repeated HTTP overhead.
✅ Server-Sent Events (SSE) provide a persistent unidirectional server-to-client stream over HTTP with built-in reconnection and event IDs.
✅ Choose WebSockets for chat, gaming, and collaborative tools; SSE for notifications and live feeds; long-polling as a compatibility fallback.
✅ WebSockets complicate load balancing because connection state is server-resident — sticky sessions or a message broker like Redis Pub/Sub is needed to route messages across nodes.
