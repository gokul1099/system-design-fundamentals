Cache Read Strategies

⏱ 4 min read 🟡 Intermediate Caching Patterns

Core Concept

Here are the two famous cache read strategies:

Read through cache

A read-through cache strategy is a caching mechanism where the cache itself is responsible for retrieving the data from the underlying data store when a cache miss occurs. In this strategy, the application requests data from the cache instead of the data store directly. If the requested data is not found in the cache (cache miss), the cache retrieves the data from the data store, updates the cache with the retrieved data, and returns the data to the application.

This approach helps to maintain consistency between the cache and the data store, as the cache is always responsible for retrieving and updating the data. It also simplifies the application code since the application doesn't need to handle cache misses and data retrieval logic. The read-through cache strategy can significantly improve performance in scenarios where data retrieval from the data store is expensive, and cache misses are relatively infrequent.

Beginner Analogy

💡
Cache read and write strategies are like different systems a librarian might use for managing book checkouts. In cache-aside, the librarian checks the shelf first and only fetches from the warehouse on a miss — the patron does the legwork. In read-through, the librarian always handles fetching transparently. In write-through, every returned book is shelved immediately in both the shelf and the catalogue. In write-behind, the librarian batches catalogue updates for later, handling returns quickly but risking the catalogue being briefly out of date.

Real-World Examples

🎬 Netflix
Netflix uses cache-aside (lazy loading) for metadata like show descriptions and ratings. The application checks the cache first; on a miss, it fetches from the database and populates the cache. This pattern keeps unused data out of the cache and is simple to implement.
💳 Stripe
Stripe uses write-through caching for payment state data to ensure that every write is immediately reflected in both the cache and the database. Given the financial nature of the data, consistency is paramount and the additional write latency is acceptable. This prevents a situation where a payment status is stale in cache.
🐦 Twitter/X
Twitter uses a read-through cache for timeline construction — the cache layer transparently fetches from the database on misses and populates itself. This simplifies application code by hiding cache population logic. Write-behind buffering is used for less critical metrics like view counts to absorb write spikes.

│ App → Cache (MISS) → DB → App→Cache  │

Interview Tips

  • 💡 Be able to name and sketch all four strategies: cache-aside, read-through, write-through, and write-behind — interviewers often ask you to compare them.
  • 💡 Cache-aside is the most commonly used in practice because it gives the application full control and prevents caching unused data.
  • 💡 Write-behind improves write performance by batching DB writes asynchronously, but risks data loss if the cache fails before flushing.
  • 💡 Discuss when to combine strategies: use read-through with write-through for strong consistency, or cache-aside with TTL for simplicity and resilience.

Key Takeaways

✅ Cache-aside loads data into the cache only on a miss, giving the application full control over caching logic.
✅ Read-through makes the cache layer responsible for fetching from the database transparently on misses.
✅ Write-through updates both cache and database synchronously on every write, ensuring consistency.
✅ Write-behind (write-back) updates the cache immediately and the database asynchronously, improving write throughput.
✅ Strategy choice depends on consistency requirements, write frequency, and acceptable risk of data loss.
