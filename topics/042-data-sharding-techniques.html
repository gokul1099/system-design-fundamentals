Data Sharding Techniques

⏱ 4 min read 🟡 Intermediate Sharding Distributed Systems

Core Concept

Data sharding, a type of horizontal partitioning, is a technique used to distribute large datasets across multiple storage resources, often referred to as shards. By dividing data into smaller, more manageable pieces, sharding can improve performance, scalability, and resource utilization. Below are several data sharding techniques with examples:

1. Range-based Sharding

In range-based sharding, data is divided into shards based on a specific range of values for a given partitioning key. Each shard is responsible for a specific range, ensuring that the data is distributed in a predictable manner.

Example: An e-commerce platform stores order data and decides to shard it based on order dates. Shards can be created for specific date ranges, such as monthly or yearly intervals. When a query targets a specific date range, only the relevant shard needs to be accessed, which improves query performance.

Beginner Analogy

💡
Data sharding techniques are like different strategies for splitting a massive library into branch libraries. You could split by subject (vertical sharding — science books in one branch, history in another). You could split alphabetically by author (range-based horizontal sharding). Or you could assign books to branches based on a hash of the ISBN (hash-based sharding), guaranteeing each branch gets roughly the same load. Each approach makes some lookups easy and others harder.

Real-World Examples

💬 Discord
Discord shards its message database by channel ID using hash-based sharding. Each channel's messages are stored on a specific shard determined by hashing the channel ID. This distributes load evenly because each channel receives independent traffic. Discord migrated to Cassandra's partition-per-channel model to handle their massive message volumes.
📁 Dropbox
Dropbox shards its metadata store by user ID, so that all metadata for a given user's files lives on one shard. This enables efficient per-user queries without cross-shard joins. Shards are sized to hold thousands of users, and heavily active users can be moved to dedicated shards to prevent hot spots.
💼 LinkedIn
LinkedIn uses directory-based sharding for its member database, maintaining a lookup table that maps member IDs to specific shard locations. When a member moves to a different shard (for load balancing), only the directory table needs updating — the application continues to work without changes. This provides flexibility at the cost of an extra lookup.

Interview Tips

  • 💡 Distinguish sharding techniques: algorithmic/hash sharding needs no external directory but requires rehashing if N changes; directory-based sharding is more flexible but adds a lookup hop.
  • 💡 Explain shard key selection criteria: high cardinality (many unique values), even distribution, and alignment with most common query access patterns.
  • 💡 Discuss resharding — the process of redistributing data when shard count changes — and why consistent hashing minimizes the data that must move.
  • 💡 Mention shard hot spots: if a shard key correlates with traffic intensity (e.g., celebrity users, trending topics), that shard becomes overloaded — mitigation includes random salting or dedicated hot-shard handling.

Key Takeaways

✅ Hash-based sharding distributes data evenly using a hash function but makes range queries and resharding expensive.
✅ Directory-based sharding uses a lookup table to map keys to shards, allowing flexible rebalancing at the cost of an extra lookup.
✅ Range-based sharding supports efficient range queries but risks creating hot shards on monotonically increasing keys.
✅ Choosing the right shard key is the most critical decision — it should distribute data evenly and align with common query patterns.
✅ Consistent hashing minimizes data movement during resharding by only remapping a fraction of keys when nodes are added or removed.
