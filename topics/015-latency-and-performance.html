Latency and Performance

⏱ 4 min read 🟢 Beginner Latency Performance Core Concepts

Core Concept

Latency and performance are critical aspects of distributed systems, as they directly impact the user experience and the system's ability to handle large amounts of data and traffic. In distributed systems, optimizing latency and performance involves considering factors such as data locality, load balancing, and caching strategies. Here's an overview of these aspects of latency and performance:

A. Data Locality

Data locality refers to the organization and distribution of data within a distributed system to minimize the amount of data that needs to be transferred between nodes. By storing related data close together or near the nodes that access it most frequently, you can reduce the latency associated with data retrieval and improve overall performance. Techniques to achieve data locality include data partitioning, sharding, and data replication.

B. Load Balancing

Load balancing is the process of distributing incoming network traffic or computational workload across multiple nodes or resources to ensure that no single node is overwhelmed. This helps to optimize resource utilization, minimize response times, and prevent system overloads. Various load balancing algorithms, such as round-robin, least connections, and consistent hashing, can be employed to achieve efficient load distribution and improved system performance.

C. Caching Strategies

Caching is a technique used to store frequently accessed data or computed results temporarily, allowing the system to quickly retrieve the data from cache instead of recalculating or fetching it from the primary data source. By implementing effective caching strategies, you can significantly reduce latency and improve the performance of your distributed system. Common caching strategies include in-memory caching, distributed caching, and content delivery networks (CDNs).

Beginner Analogy

💡
Latency is how long it takes a letter to arrive after you send it. Performance is how many letters the post office can process per hour. A fast delivery service (low latency) that can only handle 10 letters per day has poor performance. Great system performance means both low latency for individual requests AND high throughput for total request volume.

Real-World Examples

🔍 Google
Google Search targets sub-200ms response times for every query. They found that every 100ms of added latency reduces search traffic by 1%, so they invest heavily in caching, precomputation, and globally distributed data centers to minimize round-trip times.
📦 Amazon
Amazon measured that every 100ms of latency costs them 1% in sales. This data-driven insight drives their performance engineering investment, including massive edge caching, read replicas, and DAX (DynamoDB Accelerator) for sub-millisecond read latencies.
🎮 Discord
Discord targets sub-100ms message delivery latency for voice and text. They use Elixir and Erlang for their real-time messaging infrastructure because the BEAM runtime offers predictable sub-millisecond message passing with minimal garbage collection pauses.

Interview Tips

  • 💡 Know key latency numbers by heart: memory access ~100ns, SSD ~0.1ms, HDD ~10ms, network round-trip ~0.5ms (LAN) / ~100ms (cross-continent).
  • 💡 Distinguish p50 (median), p99 (99th percentile), and p999 latency — optimizing p50 while ignoring p99 means your worst users have terrible experiences.
  • 💡 Explain that caching, connection pooling, and read replicas are the primary tools for reducing latency in data-intensive systems.

Key Takeaways

✅ Latency is the time taken for a single request to complete from the client's perspective, typically measured in milliseconds.
✅ Throughput is the number of requests a system can handle per second, a measure of overall system capacity.
✅ Latency and throughput are not the same: a high-throughput system can still have high tail latency (p99) for individual requests.
✅ Caching at multiple levels (CDN, application, database) is the most impactful technique for reducing read latency.
✅ Tail latency (p99, p999) matters more than median latency for user-facing systems because it affects a meaningful fraction of users.
