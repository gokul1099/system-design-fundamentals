Partitioning Methods

⏱ 4 min read 🟡 Intermediate Partitioning Sharding

Core Concept

Designing an effective partitioning scheme can be challenging and requires careful consideration of the application requirements and the characteristics of the data being processed. Below are three of the most popular schemes used by various large-scale applications.

a. Horizontal Partitioning

Also known as sharding, horizontal data partitioning involves dividing a database table into multiple partitions or shards, with each partition containing a subset of rows. Each shard is typically assigned to a different database server, which allows for parallel processing and faster query execution times.

For example, consider a social media platform that stores user data in a database table. The platform might partition the user table horizontally based on the geographic location of the users, so that users in the United States are stored in one shard, users in Europe are stored in another shard, and so on. This way, when a user logs in and their data needs to be accessed, the query can be directed to the appropriate shard, minimizing the amount of data that needs to be scanned.

The key problem with this approach is that if the value whose range is used for partitioning isn’t chosen carefully, then the partitioning scheme will lead to unbalanced servers. For instance, partitioning users based on their geographic location assumes an even distribution of users across different regions, which may not be valid due to the presence of densely or sparsely populated areas.

Beginner Analogy

💡
Partitioning methods are like different ways to divide a school into classes. Range partitioning is like dividing by last name — A through F in one class, G through P in another. Hash partitioning is like randomly assigning students by their student ID number modulo the number of classes, ensuring each class is the same size. List partitioning is like manually assigning specific students to specific classes by hand. Each method has different trade-offs for balance and query efficiency.

Real-World Examples

🛒 Amazon
Amazon uses hash partitioning for DynamoDB, where the partition key is hashed to determine which physical partition stores the item. This ensures an even distribution of data and prevents hot partitions. For time-series data like order history, range partitioning on timestamp is used to allow efficient range queries.
🚗 Uber
Uber uses list partitioning by geographic region for city-specific operational data — NYC, SF, and London are distinct partitions. This allows Uber to comply with data residency laws (EU data stays in EU) and localize operations. Global analytics queries that need all regions must query across partitions.

Interview Tips

  • 💡 Know the three main methods: range (by sorted value ranges), hash (by hash function), and list (by explicit value enumeration).
  • 💡 Discuss range partitioning's advantage for range queries (e.g., 'all orders in the last month') but its risk of hot spots on monotonically increasing keys like timestamps.
  • 💡 Hash partitioning solves hot spots and ensures even distribution but makes range queries expensive since adjacent rows are in different partitions.
  • 💡 Explain consistent hashing as a special case of hash partitioning used in distributed systems to minimize data movement when nodes are added or removed.

Key Takeaways

✅ Range partitioning assigns contiguous ranges of key values to partitions — efficient for range queries but risks hot spots.
✅ Hash partitioning applies a hash function to keys to distribute rows evenly — prevents hot spots but makes range queries expensive.
✅ List partitioning explicitly maps specific values to specific partitions — useful for geographic or categorical distributions.
✅ Consistent hashing is a special hash partitioning method that minimizes data reshuffling when the number of partitions changes.
✅ The right partitioning method depends on the predominant query patterns and the distribution of data values.
